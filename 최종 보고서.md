### 1. 초록 (Abstract)

EcoSense는 실시간 환경 모니터링(실외·실내·수면)을 위한 클라이언트 중심 PWA + 서버 보조 백엔드 시스템이다. Web Audio API, Web Bluetooth API, IndexedDB, FastAPI, PostgreSQL, KMA·에어코리아 프록시, Brevo 이메일 인증, reportlab+matplotlib PDF 리포트를 통합하여 구현하였다.  

주요 특징은 **30일 자동 만료 정책**, **스파이크 오디오 Blob 클라이언트 전용 저장**, **이메일 인증 + 설문 기반 맞춤 PDF**, **Outdoor 캐싱(기본 TTL 10분, 환경변수 `OUTDOOR_CACHE_TTL_MINUTES`로 조정)** 등이다.  

본 보고서는 SRS v1.1, 설계문서 v1.0, PlantUML 다이어그램, API/DB 스키마, 프론트엔드 구조 및 훅 상세 문서, GitHub 저장소, Vercel/Render 배포 환경을 모두 통합하여 작성되었으며, 요구사항부터 배포까지의 전체 과정을 traceability matrix와 함께 상세히 기록한다. 테스트 보고서와 AI 활용 분석을 별도 장으로 구성하여 학술적·실무적 완성도를 높였다.

### 2. 전체 프로세스 요약 (Requirements → Design → Implementation → Testing → Deployment)

#### 2.1 요구사항 분석 단계 (SRS v1.1 기반, 2026년 5월 초)
- **주요 기능 요구사항 (FR)**: UC-001 실외 대시보드, UC-002 실내 BLE, UC-003 수면/소음 스파이크, UC-004 PDF 리포트, UC-005 인증·동기화, UC-006 설문조사.
- **비기능 요구사항 (NFR)**: 30일 데이터 만료, 클라이언트 Blob 저장, CORS·보안 쿠키, 모바일 반응형, Web Bluetooth Chrome 제한 안내.
- **사용자 설문 결과 반영**: Appendix A (부록 참조) — 실내 습도 알림, 수면 스파이크 청취, PDF 맞춤 코멘트 필요성 확인.
- **산출물**: SRS v1.1 PDF, Use Case 명세, 초기 PlantUML UC 다이어그램.

#### 2.2 설계 단계 (2026년 5월 중순, 설계문서 v1.0)
- **아키텍처**: Client-Centric + Server-Assisted (Presentation: React 18 + Vite + Tailwind, Client Data: IndexedDB + Zustand, Backend: FastAPI + PostgreSQL on Render, Frontend: Vercel).
- **주요 설계 결정**:
  - 오디오 Blob → 서버 미저장 (Render Ephemeral FS 회피) → IndexedDB + expires_at 30일.
  - Outdoor → location 키 기반 공용 캐시 테이블 + TTL (OUTDOOR_CACHE_TTL_MINUTES).
  - Auth → Brevo + verify_code + is_verified 플래그.
  - Session/Spike 통합 모델 (type: OUTDOOR/INDOOR/SLEEP).
- **산출물**: 설계문서 PDF, PlantUML 전체(아키텍처, 클래스, 6개 UC 시퀀스), db_schema.txt, API Endpoint + JSON Schema, Frontend Structure.txt, useAudio/useBluetooth/useSync 설계 요약.

#### 2.3 구현 단계 (2026년 5월 하순 ~ 6월 중순, v1.0 → v1.2)
- **Frontend (Vercel, ecosense-sooty.vercel.app)**: pages/ (Start, Register with verify, Survey, Outdoor/Indoor/Sleep/Reports/Settings), hooks/ (useAudio, useBluetooth, useSync), stores/ (Zustand), services/ (api.ts with refresh), IndexedDB v2 (sessions + spikes store).
- **Backend (Render)**: FastAPI 라우터, SQLAlchemy + raw DDL (user_preferences, user_feedback), Outdoor Service (KMA 초단기 → ASOS 폴백 + AirKorea + Nominatim), Report Service (BytesIO 스트리밍 PDF), Sync upload/download, cleanup endpoint.
- **핵심 구현 포인트** (첨부 파일 직접 반영):
  - BLE: LYWSD03MMC UUID + getInt16/getUint8 파싱 + battery 계산 공식.
  - Audio: AnalyserNode fftSize 8192, RMS dB 계산, 측정 시작 후 3초 배경소음 캘리브레이션, 배경+15dB 초과 시 스파이크 시작, 배경+8dB 이하 1초 연속 시 종료하는 히스테리시스 구조, MediaRecorder webm/opus + IndexedDB saveSpikeWithBlob 저장.
  - Sleep Analysis: 90분 수면 사이클을 기준으로 얕은 수면(N1/N2), 깊은 수면(N3), REM 수면을 시간대 기반 휴리스틱 hypnogram으로 추정하고, 각 수면 단계별 소음 스파이크 분포를 분석.
  - DB/Storage: PostgreSQL 스키마 (cascade FK, 30일 cleanup), outdoor_data_cache TTL, 오디오 Blob은 서버 미저장 및 클라이언트 IndexedDB 전용 보관.
  - Report: ReportLab + matplotlib 기반 PDF 생성, Noto Sans KR 번들 폰트 등록으로 그래프 내부 한글 깨짐 방지, 설문 결과 기반 맞춤 코멘트 반영.
  - v1.2 신규: 이메일 인증 3단계(register/verify/resend), Survey API, Report history, force_refresh, weather_source 필드.
- **GitHub 현황**: backend/, frontend/ 및 주요 설계·API·DB·구조 문서 관리, 기능별 커밋 이력 유지.

#### 2.4 배포 및 운영 단계
- **Frontend**: Vercel (VITE_API_BASE_URL = Render URL).
- **Backend + DB**: Render Web Service + PostgreSQL (env: DATABASE_URL, BREVO, KMA keys, COOKIE_SECURE=true, RESET_DB_ON_STARTUP).
- **보안**: HttpOnly Secure Cookie, CORS_ORIGINS 제한, SECRET_KEY 32바이트.
- **데이터 정책**: 로그아웃은 계정 세션만 종료하며, 이 기기의 수면 측정·녹음 데이터는 IndexedDB에 유지된다. 서버에는 세션·스파이크 메타데이터만 동기화하고, 오디오 Blob은 클라이언트 로컬에만 저장한다.
- **운영 기능**: /health, /, POST /sessions/cleanup (필요 시 호출), 30일 자동 정리.
- **현재 상태**: Vercel + Render 모두 정상 배포 완료, 실시간 테스트 가능 ([ecosense-sooty.vercel.app](https://ecosense-sooty.vercel.app)).

#### 3.4 테스트 커버리지 결과 (요약)

현재 프로젝트는 정량 커버리지 수치보다 **핵심 사용자 흐름의 동작 검증**을 우선하였다. 자동 커버리지 리포트는 아직 CI에 고정 산출물로 포함하지 않았으므로, 본 보고서에서는 임의의 퍼센트 수치 대신 검증 범위를 명시한다.

#### 3.5 발견된 버그 및 해결 로그 (대표 사례)

1. **Bug-2026-05-28**: BLE disconnect 시 characteristic event listener leak → 메모리 누수  
   **해결**: disconnect() 함수에 removeEventListener + null 초기화 추가 (useBluetooth.ts v2)

2. **Bug-2026-06-02**: PDF BytesIO 스트리밍 시 Render Ephemeral FS 경고  
   **해결**: 디스크 쓰기 제거, Response(BytesIO.getvalue(), media_type="application/pdf")

3. **Bug-2026-06-05**: /auth/verify 후 is_verified=true 미반영 (트랜잭션 누락)  
   **해결**: DB commit + atomic transaction 적용

4. **Bug-2026-06-07**: IndexedDB cleanupExpiredSpikes()와 세션/스파이크 보관 정책이 문서에 명확히 구분되지 않음  
   **해결**: spikes는 30일 만료 및 로컬 초기화 대상, sessions는 기록 조회용으로 별도 관리한다는 정책을 문서화

5. **Bug-2026-06-10**: CORS_ORIGINS에 Vercel Preview URL 누락 → 401 on login  
   **해결**: .env에 *.vercel.app 추가 + restart

6. **Bug-2026-06-12**: 50dB 수준의 지속 배경소음 환경에서 스파이크 녹음이 종료되지 않는 문제  
   **해결**: 3초 배경소음 캘리브레이션, backgroundRef 기반 최신값 참조, 시작/종료 임계값을 분리한 히스테리시스 적용

7. **Bug-2026-06-15**: PDF 보고서 그래프 내부 한글 깨짐(Render Linux 환경에서 matplotlib 기본 폰트가 한글 미지원)  
   **해결**: Noto Sans KR 폰트를 프로젝트에 번들하고, ReportLab/matplotlib 양쪽에 공통 폰트 탐색·등록 로직 추가

### 4. AI 활용 분석 (Cursor AI, 2026년 6월 배포·디버깅 단계)

본 프로젝트는 각 기능의 1차 설계 및 구현을 담당 팀원들이 주도적으로 수행하였으며(부록 G 참조), Cursor AI는 시스템 통합, 배포 직전 디버깅, 그리고 불일치 문서 동기화 등의 보조 도구로 집중 활용하였다. 아래 내용은 Cursor 에이전트 세션의 실제 대화 기록을 바탕으로 정리한 협업 분석이다.

#### 4.1 Cursor AI가 기여한 주요 작업 (실제 세션 기준)

**A. 배포·운영 디버깅**
- Vercel + Render + PostgreSQL 배포 순서 및 env 변수(`SECRET_KEY`, `CORS_ORIGINS`, `COOKIE_SECURE`, `DATABASE_URL`) 점검
- Render `Port scan timeout`, `RESET_DB_ON_STARTUP`로 인한 startup 지연 원인 분석
- CORS 오류가 실제로는 **500 Internal Server Error**를 가리는 경우(register/login) 진단
- `frontend/src/services/api.ts` URL 정규화로 `/api/api/...` 이중 경로·405 문제 해결
- PostgreSQL `sqlalchemy f405`(스키마 불일치) 원인 분석 및 `RESET_DB_ON_STARTUP` 1회 사용 가이드

**B. 실외 환경(Outdoor) 개선**
- 에어코리아 API 키 연동 확인, 하드코딩 측정소명(`창원`) 문제 진단
- `airkorea_station.py` 추가: 전국 측정소 WGS84 캐시 + 최근접 측정소 자동 선택
- 기상청(기온·강수·SKY) + 에어코리아(PM2.5/PM10) 역할 분리 및 전국 지역 대응
- GPS 좌표 → 사람이 읽을 수 있는 지역명 표시(`resolve_display_location`)
- mock 캐시 저장 제거, `is_mock`/`weather_source` 표시 정책 정리
- `getUltraSrtNcst`에 SKY 코드가 없어 “맑음”만 나오던 문제 → `getVilageFcst` SKY 보완
- `OutdoorPage.jsx` UV `String(...).toFixed` 버그, `location/search` 500 + Nominatim 실패 시 로컬 fallback
- `OutdoorChart.jsx` 첫 탭을 `기온/습도` → `기온`만 표시하도록 수정

**C. 인증·이메일(Brevo)**
- “인증 전 가입·로그인 가능” 흐름 문제 분석 (register 시 DB 저장, login이 `is_verified` 미검사)
- `register_or_resend()` 도입: **미인증 계정 재가입 허용**, 인증 완료 계정만 409
- login 시 `is_verified=false` → **403** 반환, `LoginPage`에서 인증 화면으로 이동
- Brevo 발송 실패 시 503 반환·로그 기록 (`email.py`, `auth_service.py`)
- 전화번호 숫자만 입력하도록 프론트/백엔드 정규화

**D. 수면·소음 측정**
- 배경 소음 고정·스파이크 녹음 미종료 원인 분석 (stale closure, 임계값 경계, 스무딩)
- `useAudio.ts` 개선: 3초 캘리브레이션, `backgroundDbRef`, 시작/종료 히스테리시스(+15/+8), 피크 dB 저장
- `sleepAnalysis.js` 개선: 90분 사이클 REM/NREM **통계 기반 hypnogram** 추정, 단계별 스파이크 분석
- `SleepAnalysisResult.jsx`에 수면 단계 분포 UI 추가
- 로그아웃 후 로컬 기록 유지 정책 분석 및 설정/수면 페이지 안내 문구 추가

**E. 보고서·문서**
- PDF 그래프 한글 깨짐(Render Linux + matplotlib) 원인 분석
- `NotoSansKR-Regular.otf` 번들 + `report_service.py` 공통 폰트 탐색 로직 추가
- `API Endpoint.txt`, `db_schema.txt`, `Diagram PlantUML.txt`, `useAudio.txt` 등 **현재 코드와 불일치 문서 동기화**
- 본 `최종 보고서.md`의 과장된 테스트 수치·구버전 설명 검토 및 수정

**F. 테스트·검증 (AI가 설계·실행 보조)**
- `backend/scripts/e2e_api_test.py`: auth, outdoor, sync, report download 전체 흐름
- `backend/scripts/outdoor_smoke_test.py`, `test_airkorea.py`: 실외·에어코리아 연동 확인
- 로컬 uvicorn + Vite 실행 후 CORS·실데이터·지역 검색 수동 확인
- Render 배포 URL에 대한 curl/httpx POST register 재현 테스트

#### 4.2 AI 활용의 장점 (Pros)

- **배포 환경 이슈 대응 속도**: CORS/500/DB 스키마/Render startup 같은 “로컬에서는 안 터지는 문제”를 빠르게 좁혀 원인 후보를 제시
- **외부 API 연동 가속**: 에어코리아·기상청 기술문서를 읽고, 하드코딩 측정소 → 동적 최근접 측정소 구조로 전환하는 설계·코드 초안 제공
- **버그 원인 설명**: 수면 측정의 stale closure, 이메일 인증 흐름 불일치, PDF 한글 깨짐 등 **현상→코드→수정 방향**을 한 번에 정리
- **문서·보고서 보조**: 코드 변경 후 `.txt` 설계 문서와 최종 보고서를 현재 구현에 맞게 맞추는 작업 부담 감소
- **테스트 스크립트 초안**: smoke/e2e 스크립트와 검증 시나리오를 빠르게 작성 (단, 실행 결과는 사람이 확인)

#### 4.3 AI 활용의 단점 및 대응 (Cons)

- **Hallucination / 구버전 참조**: 예전 DB 스키마, 존재하지 않는 API(`setBackgroundDb`, `getCurrentDb`)를 테스트 예시에 넣는 경우 발생 → **항상 실제 파일 기준으로 재검증**
- **과장된 수치 생성**: “커버리지 84%”, “48개 테스트 100% Pass”처럼 **실제 CI 산출물 없이 쓰면 위험** → 본 보고서에서는 제거하고 “수동·스크립트 검증”으로 표현
- **배포·비밀값은 사람이 최종 확인**: Render env, Brevo sender verified, API 키 종류(기상청 vs 에어코리아)는 AI가 틀릴 수 있어 **팀원이 대시보드·로그로 확인**
- **대응 원칙**: “AI 초안 → 코드/로그로 사실 확인 → 수정 → 로컬/배포 URL에서 재테스트 → 문서 반영”

#### 4.4 AI 협업 프로세스 (본 프로젝트에서 실제 사용한 방식)

```text
1) 문제 발생 (배포 URL, 콘솔 오류, 기능 이상)
2) Cursor에 증상 + Render/Vercel 로그 + 관련 파일 전달
3) AI가 코드 경로 추적 및 원인 가설 제시
4) 팀원이 가설 검증 (curl, e2e 스크립트, 브라우저 Network 탭)
5) AI가 수정 패치 작성 → 사람이 리뷰 후 merge/배포
6) 문서(.txt/.md, 최종 보고서)를 현재 구현과 동기화
```

**종합 평가**: Cursor AI는 **초기 기능 구현**보다 **배포 후 통합 디버깅·외부 API 연동·정책 정리·문서화** 단계에서 특히 유용했다. 다만 **최종 동작 보장과 배포 설정은 팀원의 로그·테스트 확인 없이는 불가**하며, AI 출력은 “초안 + 검증”으로 사용하는 것이 적절하다.

- **팀 협업 및 AI 활용**: GitHub 기반 협업, Vercel/Render 배포, AI 보조 설계·문서화 활용

- 실생활에서 사용할 수 있는 **PWA 형태의 환경 모니터링 서비스** 제작 (게스트/로그인 모두 지원).
- **배포 환경을 고려한 설계** (Client-Centric + Server-Assisted, Ephemeral FS 대응, TTL 캐시, Blob 전략).
- **학술적 가치**: Software Engineering 수업 요구사항(Use Case, Sequence, Traceability, Test Report, AI Collaboration)을 실제 구현물과 연결하여 정리.

**교훈**  
- 클라이언트-서버 데이터 분리 전략의 중요성 재확인.  
- AI는 “초안 + 검증” 조합일 때 가장 강력하다.
