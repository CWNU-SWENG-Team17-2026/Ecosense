### EcoSense 아키텍처 개요

#### 1. 전체 시스템 아키텍처

EcoSense는 **클라이언트 중심 + 서버 보조** 구조로 설계한다. 실시간 오디오 처리와 센서 제어는 브라우저에서 직접 수행하며, 백엔드는 인증, 데이터 메타데이터 영속화, PDF 생성, 외부 API 프록시 역할을 담당한다.

```
[브라우저 (PWA)]
├── Presentation Layer (React 18 + Vite + Tailwind CSS)
│     ├── Web Audio API Pipeline
│     │     ├── AnalyserNode (실시간 dB 모니터링)
│     │     └── MediaRecorder (스파이크 구간 webm/opus Blob)
│     ├── Web Bluetooth API (Xiaomi Mijia 센서)
│     └── MediaDevices API (마이크 접근)
│
├── Client Data Layer
│     ├── LocalStorage (설정 및 상태)
│     └── IndexedDB
│           ├── 게스트 측정 데이터
│           └── 수면 스파이크 오디오 Blob (로컬 저장)
│
└── HTTPS + JWT (HttpOnly Secure Cookie)
         │
[Render - Backend]
├── FastAPI
│     ├── Auth Service
│     ├── Sync Service (메타데이터 동기화)
│     ├── PDF Report Service (reportlab + matplotlib)
│     ├── Survey Service (user_preferences / user_feedback)
│     └── Outdoor Proxy (KMA 초단기/ASOS + 에어코리아 + Nominatim + DB 캐시 TTL 10분)
│
[Data Layer]
└── PostgreSQL
      └── User, MeasurementSession, Spike, OutdoorDataCache, ReportHistory, user_preferences, user_feedback
```

#### 2. 레이어 상세 설명

**1) Presentation Layer (Frontend)**
- **기술 스택**: React 18 + Vite + Tailwind CSS + Chart.js
- **배포**: Vercel
- **주요 컴포넌트**:
  - **Web Audio API Pipeline**
    - `AnalyserNode`: 실시간 데시벨 모니터링
    - `MediaRecorder`: 스파이크 구간 webm/opus Blob (임계값 이하 1초 연속 시 종료)
  - 5~10초 주기로 UI 갱신 및 필요 시 서버 메타데이터 동기화
  - Web Bluetooth를 통한 실내 센서 데이터 수신

**2) Client Data Layer**
- **IndexedDB**에 **수면 스파이크 음성 데이터(Blob)**를 직접 저장
- 로그인 사용자도 스파이크 오디오 Blob은 클라이언트에 보관하고, 서버에는 메타데이터만 동기화
- **30일 경과 데이터 자동 삭제** 로직 적용 (클라이언트 측 만료 처리)

**3) Business Logic Layer (Backend)**
- **기술 스택**: FastAPI (Python)
- **배포**: Render.com
- 주요 역할: 인증(이메일 인증), 설문, 메타데이터 동기화, PDF 생성, 기상/대기질 API 프록시 + DB 캐시

**4) Data Layer**
- **PostgreSQL** (Render): 로그인 사용자의 영속 데이터 저장
  - 주요 엔티티: `User`, `MeasurementSession`, `Spike`, `OutdoorDataCache`, `ReportHistory`, `user_preferences`, `user_feedback`
  - **스파이크 음성 오디오 파일 자체는 저장하지 않음** (클라이언트 IndexedDB 보관)
- **DB 캐시**: outdoor_data_cache 테이블, TTL 기본 10분 (OUTDOOR_CACHE_TTL_MINUTES)

#### 3. 인증 토큰 관리

- JWT는 **HttpOnly Secure Cookie** 방식으로 관리하여 XSS 공격 위험을 최소화한다.
- (LocalStorage 저장 방식은 보안상 지양)

#### 4. 주요 데이터 흐름 (수면 모드 스파이크)

1. `AnalyserNode`로 실시간 데시벨 모니터링
2. 기준 충족 시 `MediaRecorder`로 오디오 Blob 생성 (webm/opus)
3. Blob은 **IndexedDB에 로컬 저장**
4. 스파이크 메타데이터는 **측정 종료 시** 또는 설정에서 sync 시 서버 업로드
5. 30일 경과 데이터는 클라이언트/서버 각각 자동 삭제 처리

#### 5. 배포 구조

- Frontend: Vercel
- Backend + PostgreSQL: Render.com
- JWT: HttpOnly Secure Cookie

#### 6. 제약 사항 및 설계 판단

- **데이터 보관 정책 (SRS FR-S004 반영)**:  
  측정 데이터(IndexedDB의 스파이크 Blob 포함)는 **30일 경과 시 자동 삭제**된다. IndexedDB는 클라이언트 `cleanupExpiredSpikes()`, PostgreSQL은 앱 시작 시 `cleanup.py` 및 `POST /api/sessions/cleanup`으로 처리한다.
- **스파이크 음성 저장**: Render 무료 플랜의 Ephemeral File System 제약과 서버 부하를 고려하여, 오디오 Blob은 클라이언트 IndexedDB에 저장하고 메타데이터만 서버에 동기화하는 구조를 채택함.
- **Web Bluetooth**: Chrome/Edge 전용 → UI에 안내 예정
